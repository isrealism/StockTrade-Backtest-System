# 产品需求设计（功能需求）

## 1. 系统通用实体定义

本模块定义了系统内部各组件之间交互的标准数据格式，确保信号、订单、持仓和交易记录在流转过程中的一致性与完整性。

### 1.1 买入信号 (BuySignal)

- **定义**：由选股策略模块产生的原始买入建议。
- **核心要素**：股票代码、信号产生日期、策略名称及昵称、打分
- **后续可扩展**：触发信号时各项关键指标数据（用于后续分析）。

### 1.2 持仓 (Position)

- **定义**：账户当前持有的某一仓股票的资产状态。
- **核心要素**：
    - 股票代码
    - 建仓日期
    - 建仓价格
    - 仓位大小
    - 建仓成本
    - 持仓以来最高收盘价、对应时间
    - 持仓以来最低收盘价、对应时间
    - 股价最高价、对应时间
    - 股价最低价、对应时间
    - 持仓天数
    - 买入策略、买入时的指标数据（根据code关联相应的BuySignal）
    - 该仓位的初始市值、当前市值
    - 未实现的利润（Unrealized PnL）及其百分比 (Unrealized PnL Percentage)：`shares x current_price - cost_basis`
- **支持操作**：
    - 最高价跟踪（Highest Price Since Entry）
    - 持仓天数更新

### 1.3 订单 (Order)

- **定义**：管理交易订单状态及属性。
- **核心要素**：
    - 股票代码、执行动作（买或卖）、shares、信号产生日期、执行日期
    - 如果订单执行成功，标记订单状态为已执行，记录：
        - 状态：默认为 PENDING
        - 执行价格
        - 佣金
        - 印花税（买入为0.0）
        - 滑点
        - 总成本
        - 总收益
        - 卖出订单记录卖出原因
        - 买入订单记录选股策略
- **状态管理**：支持 PENDING (待处理) → EXECUTED (已成交) / FAILED (废单) / CANCELLED (撤单) 的状态流转。

### 1.4 交易记录 (Trade)

- **定义**：一笔完整的"买入+卖出"明细记录。
- **核心要素**：
    - 股票代码
    - 建仓日期、建仓价格、shares、建仓成本
    - 清仓日期、价格、收益
    - （Optional）买入策略、清仓原因
    - 收益记录（自动计算）：
        - 毛利润（Gross PnL）：`shares × (exit_price − entry_price)`
        - 毛利润百分比（Gross PnL Percentage）：`(exit_price − entry_price) / entry_price`
        - 净利润（Net PnL）：`exit_proceeds − entry_cost`
        - 净利润百分比（Net PnL Percentage）：`(exit_proceeds − entry_cost) / entry_cost`
    - 持仓天数
    - 最大未实现利润百分比（Max Unrealized PnL Percentage）
- 将核心要素转为字典形式方便统一管理。

## 2. 仓位与资金管理

### 2.1 功能概述

本模块负责管理账户中的持仓（Position）、资金（Cash）以及交易中订单资金的冻结与解冻，保证交易在严格的资金约束和仓位约束下进行，资金永不为负。遵守 T+1 日交易制度，每日更新持仓状态，并执行新的买卖订单，从账户中扣除 / 冻结相应资金。支持随时调用查看账户中的可用资金、仓位明细、持仓状态、被冻结资金、待释放资金等，支持获取资产曲线和交易记录明细。

### 2.2 账户资产结构

- 总资产 = 持仓市值 + 总现金
- 持仓市值 = 所有持仓中的股票市值 x shares
- 总现金 (Cash) = 可用资金 (available_cash) + 冻结资金 (frozen_cash)

### 2.3 仓位管理

#### 2.3.1 更新持仓

每日收盘后，遍历所有持仓股票：

- 更新持仓天数（days_held + 1）
- 更新期间最高价（记录自建仓以来的最高收盘价/最高价）
- 例外处理：若当日停牌，则不更新价格与持仓天数

#### 2.3.2 更新资产曲线

每日收盘结算后，计算当日总资产，并向 equity_curve 列表追加记录：

- 包含：日期、总现金、持仓市值、总资产、持仓数量、冻结资金、待释放资金。

#### 2.3.3 开新仓

判断当日能否开新仓，如果当前已满仓或可用资金不够目标仓位，则拒绝所有买入信号，次日不开新仓。

**1. 计算目标仓位规模**

- 加载买入信号列表，将买入信号按 Score 排序
- 计算明日开盘最大可用资金：`Projected_Cash_T+1 = Cash_now + Settling_Funds_T+1`
    - `Cash_now`：T 日收盘后的当前可用现金。
    - `Settling_Funds_T+1`：即将在 T+1 日开盘前到账的资金（即 T 日卖出成交产生的待结算金额）。
- 资金分配流程：
    - 初始化 `Remaining_Cash = Projected_Cash_T+1`
    - 按排序依次遍历每个信号：
        - 满仓检查：如果 (当前持仓数 + 已生成订单数) >= 最大持仓限制，终止分配流程。
        - 如果该标的已在持仓或已在待执行列表中，跳过。
        - 依据仓位分配逻辑，计算该信号的目标仓位金额 `Target_Cost`。
        - 资金条件判断：
            - 加入资金安全垫（Buffer）控制，计算时使用 `Cash_now = 98% × Cash_now` 以防次日开盘价过高出现资金不够买入的情况。
            - 如果 `Remaining_Cash >= Target_Cost`：
                - 根据 `shares = Target_Cost / Close_T` (round down to closest 100)，返回 shares
                - 调用模拟交易执行模块的函数计算预估成本 `Target_Total_Cost`
                - `Remaining_Cash -= Target_Total_Cost`
                - 将订单加入待执行队列
            - 若资金不足：
                - Shares 返回 0，尝试队列中的下一个（寻找更低价的标的）

**2. 仓位分配逻辑**

- **等权分配**：目标仓位 = 当前总资产 / 最大持仓数
- **基于风险分配**：
    - 如果当日及之前的数据长度不够14天或 ATR 无法计算，则自动使用等权分配
    - `shares = 单个仓位可以承受的亏损金额 / (2倍ATR) = (initial_capital × 0.01) / (2 × ATR)`
    - `TR = max(High_T − Low_T, |High_T − Close_T-1|, |Low_T − Close_T-1|)`
    - 修正第一个交易日的 TR 为当日振幅；
    - `ATR` = 从当日往前 period 个交易日的平均TR（period 可配置，默认为 14）

#### 2.3.4 产生订单

**1. 产生买入订单**

- 检查能否开新仓，如果不能，则不产生新订单。
- 检查该股票是否已经在持仓，如果已经持仓，则不产生新订单
- 计算目标仓位，如果目标仓位为 0，则不产生新订单
- 该订单执行日期 = 信号产生日期 + 1 交易日
- 生成买入订单，订单应包含：股票代码、订单行为（买）、目标仓位 shares、买入信号产生日期、订单执行日期、选股器
- 将该订单加入待执行订单队列。

**2. 产生卖出订单**

- 检查是否正在持仓，如果没有持仓，则不产生订单
- 检查是否可以卖出（当天买入的股票不可以卖出），如果不可以，不产生订单
- 订单执行日期 = 信号产生日期 + 1 交易日
- 生成卖出订单，应包含：股票代码、订单行为（卖）、仓位大小 shares、卖出信号产生日期、订单执行日期、卖出原因、买入时对应的选股器
- 将该订单加入待执行订单队列。

### 2.4 资金追踪

- **资金冻结规则**：
    - T日产生卖单，T+1 日开盘执行卖单后，将收益所得冻结，当日无法使用，T+2 日解冻这笔资金加入现金账户，用于后续再投入。在管理冻结资金时，应以解冻日期进行管理，以便到期释放。

## 3. 模拟交易执行模块

### 3.1 功能概述

本模块作为回测系统的执行引擎，负责将 T 日生成的订单转化为 T+1 日的实际成交。其核心职责是在严格遵循 A 股 T+1 制度 的前提下，利用 T+1 日开盘数据执行订单，自动处理交易风控（如停牌、涨跌停拦截）与成本清算（扣除税费、滑点、佣金）。

### 3.2 订单执行规则

- 如果当日该股停牌，不允许执行，返回False
- 如果该股开盘即涨停或跌停，则不允许执行，返回False并输出原因。

### 3.3 订单执行

- 检查订单执行日期是否为当日，如果不是，则将该订单放回待执行订单队列
- 检查该股票在当日是否正常开市，如果没有，则该订单作废，标记为失败，加入已执行订单行列
- 数据规范性检查
- 检查是否可以执行订单（4.1 判断），如果不能执行，则订单作废，记录订单失败原因，加入已执行订单行列
- 以开盘价买入/卖出，执行订单
- **如果是买入订单**：
    - 支持根据输入的价格和股数计算某一只目标股票的预估成本
    - 检查现有资金能否买入，如果不能，输出提示并作废订单
    - 将买入资金从现金 (cash) 中扣除：订单 `total_cost = 当日开盘价 x shares + 滑点(slippage) + 佣金(commission)`
    - 建立仓位，应包含：股票代码、建仓日期、建仓价格、仓位大小、买入资金、选股器
    - 将该仓冻结，当日不允许卖出，T+1 日开始解冻，标记为可以卖出
- **如果是卖出订单**：
    - 计算最大未实现收益：`(highest_price_since_entry − entry_price) / entry_price`
    - 将该订单记录入交易记录，应包含信息：股票代码、建仓日期、建仓价格、仓位大小、买入资金、选股器、清仓日期、清仓价格、收益、清仓原因、持仓周期、最大未实现收益
    - 订单收益 `net_proceeds = 当日开盘价 x shares - 滑点 - 佣金 - 印花税(stamp tax)`。执行卖出后将收益冻结，T+1 日可解冻用于后续买入
    - 将该股从持仓中删除
- 将该订单状态标记为已执行 EXECUTED

### 3.4 交易规则约束

- 支持获取指定股票的涨跌停限制（如科创板20%，主板10%）
- A股市场以 100 股为最小交易单位
- 买入和卖出订单的执行都在 T+1 日开盘时，因此计算资金时统一使用 T+1 日开盘价
- 如果目标股票在执行当日停牌或开盘涨跌停，则无法进行任何操作，该订单自动作废


## 4. 信号质量过滤与换仓模块

### 4.1 功能概述

本模块在原有选股器产生信号之后、生成买入订单之前，引入两层新机制：

一是基于历史 score 分布的百分位过滤，只允许质量高于历史水位的信号进入订单生成阶段；

二是基于信号质量比较的主动换仓逻辑，当持仓中存在亏损较重且入场质量明显偏弱的仓位时，用当日更优质的信号替换它，提升资金使用效率。

两个机制均为可选功能，通过配置参数独立开关，默认关闭，不影响原有回测流程。

### 4.2 信号 Score 历史百分位过滤

#### 4.2.1 设计目标

选股器每日产生的信号数量可能波动较大，部分信号虽满足技术形态条件，但综合评分偏低。若无约束地将所有信号转为订单，会导致仓位被低质量信号占用。本模块通过维护一份动态的历史 score 分布，设定百分位阈值，仅保留评分处于历史强势区间的信号。

#### 4.2.2 历史 Score 分布的维护

- 系统维护一个全局 `score_history` 列表，按时间顺序追加每日所有原始信号的 score（过滤前追加，保证今日 score 不参与今日阈值计算，防止未来函数）。
    
- 采用"预热期预填充"方案（Bootstrap）：在回测正式开始前，利用已加载的 `lookback_days` 预热数据，完整跑一遍选股器管线，将产生的所有信号 score 填充进 `score_history`。
    
    - 预热期运行完整的选股器逻辑，包括组合模式（AND/OR/TIME_WINDOW 等），确保预热期信号与主循环口径完全一致。
    - 预热期运行时压制日志输出，不影响主循环日志可读性。
    - 预热完成后，输出当前分布摘要：p25、p50、目标百分位值、p75、p90。
- 若预热期内某日选股器运行出错，跳过该日，继续预热，不中断流程。
    
- 主循环每日将当日信号 score 追加进 `score_history` 后再执行过滤，保证历史分布随回测推进持续更新。
    

#### 4.2.3 过滤逻辑

**每日信号处理流程如下：**

a. 获取当日所有选股器产生的原始信号，按 score 从高到低排序，形成原始信号池。

b. 将原始信号池中所有 score 追加进 `score_history`（此时尚未过滤）。

c. 计算当日 score 过滤阈值：取 `score_history` 中的第 `score_percentile_threshold` 百分位数。

d. 仅保留 `score ≥ 阈值` 的信号，过滤后信号仍保持 score 降序排列，进入后续买入/换仓决策。

e. 输出日志：当日阈值、过滤前信号数、过滤后信号数。

- 若 `score_history` 数量不足 `score_min_history` 条（理论上预热后不会出现，但作为保障）：退化为不过滤，放行所有信号，并输出警告日志。

#### 6.2.4 参数配置

- **score_filter_enabled**（布尔，默认 False）：总开关，False 时跳过全部过滤逻辑。
- **score_percentile_threshold**（浮点，默认 60.0）：百分位阈值，例如 60.0 表示只保留历史前 40% 的强信号。
- **score_min_history**（整数，默认 30）：触发百分位计算的最低历史样本数，不足时不过滤。

### 6.3 主动换仓逻辑 (Rotation)

#### 6.3.1 设计目标

在持仓数已满或接近满仓时，若当日出现评分显著优于某亏损仓位入场时评分的新信号，系统应主动替换该仓位，而非因满仓而放弃高质量入场机会。换仓同时生成对应的 T+1 卖出订单和买入订单，遵守 T+1 交易制度。

#### 6.3.2 换仓判断流程

每日在卖出决策完成之后、正常买入订单生成之前，执行以下步骤：

**1. 入场 score 持久化**

- 每笔买入成交后，将该信号的 score 写入对应 Position 的 `buy_signal_data` 字段（key: `entry_score`），作为后续换仓比较的基准。
- 换仓产生的买入订单同样在成交后写入 `entry_score`，保证所有仓位均有历史评分可查。

**2. 筛选亏损候选仓位**

- 遍历当前所有持仓，排除已被本日正常卖出逻辑触发的仓位（这些仓位已在卖出队列中，不参与换仓）。
- 计算每个持仓当日收盘的 Unrealized PnL %。
- 仅保留满足 `Unrealized PnL % < −min_stop_threshold` 的仓位作为换仓候选。
- 候选仓位按亏损幅度从大到小排列（亏损最重的优先考虑）。

**3. 配对匹配**

- 按序遍历候选仓位，对每个亏损仓位寻找当日最强的、尚未被配对的信号作为替换目标。
- **匹配成功的条件（两个条件须同时满足）**：
    - **相对门槛**：`新信号 score ≥ 仓位入场 score × score_ratio_threshold`
    - **绝对门槛**：`新信号 score − 仓位入场 score ≥ min_score_improvement`
- 绝对门槛的引入是为了覆盖双边换仓摩擦成本（约 0.4%），防止低分区的微小差距触发不必要的换仓。
- 每日配对总数不超过 `max_rotations_per_day`。
- 配对结果：(亏损仓位, 替换信号) 的有序列表。

**4. 处理无 entry_score 的老仓位**

- 若持仓的 `buy_signal_data` 中不存在有效的 `entry_score`（如早期建仓时 score 尚未实现），根据 `no_score_position_policy` 参数决定行为：
    - **skip**（默认）：跳过该仓位，不参与换仓比较，行为保守。
    - **allow**：将 `entry_score` 视为 0，几乎必然满足换仓条件，行为激进。
    - **mean**：将 `entry_score` 用当前历史 score 均值填充，行为中性。

#### 6.3.3 换仓订单生成

对每一对配对结果，依次执行：

1. **生成换出仓位的 T+1 卖出订单**，订单卖出原因格式为 `rotation_replace:<替换目标代码>|score_gain=<分差>`，方便后续绩效归因。
    
2. **生成换入信号的 T+1 买入订单**：
    - 跳过持仓数量上限检查（`can_open_new_position`）——因为对应的卖出订单已生成，T+1 执行时槽位将释放，但仍检查资金是否充足。
    - 仓位大小计算沿用现有仓位分配逻辑（等权或风险敞口）。
    
3. 若买入订单生成失败（资金不足等），**自动回滚已生成的卖出订单**，保证换仓配对的完整性，不产生"只卖不买"的残缺状态。
    
4. 已完成换仓配对的信号从当日信号池中移除，不再参与后续正常买入逻辑，避免重复使用同一信号。
    
- 每对换仓操作完整记录进换仓日志（**RotationLog**），包含：
    - 日期、换出代码、换出时浮亏、换出入场 score、换入代码、换入 score、score 绝对提升、score 相对倍数、双边 shares。

#### 6.3.4 参数配置

- **rotation_enabled**（布尔，默认 False）：总开关。
- **rotation_min_stop_threshold**（浮点，默认 0.05）：触发换仓考察的最低亏损幅度，例如 0.05 表示亏损超过 5% 才考虑。
- **rotation_max_per_day**（整数，默认 2）：每日最多执行的换仓对数。
- **rotation_score_ratio**（浮点，默认 1.2）：相对门槛，新信号 score 须为旧 entry_score 的多少倍。
- **rotation_min_score_improvement**（浮点，默认 10.0）：绝对门槛，新信号 score 须高出旧 entry_score 多少分。
- **rotation_no_score_policy**（字符串，默认 skip）：无 entry_score 仓位的处理策略，支持 skip / allow / mean。

### 6.4 主循环时序更新

引入本模块后，每日主循环的执行时序调整如下：

1. 开盘前资金结算（T+1 Settlement）（不变）
2. 执行待处理订单（不变）
3. 更新持仓状态（不变）
4. 执行卖出决策，记录已触发卖出的股票代码集合（不变）
5. 获取当日原始买入信号（选股器全量输出）（不变）
6. **【新增 5.5a】Score 历史追加与百分位过滤**：将当日原始信号 score 追加进 `score_history`，计算阈值，过滤出当日有效信号池。
7. **【新增 5.5b】换仓决策**：基于有效信号池与当前持仓，执行换仓配对，生成换仓卖出与买入订单，并从有效信号池中移除已用于换仓的信号。
8. 生成正常买入订单（基于换仓后剩余的有效信号池）（逻辑不变，但输入信号池已更新）
9. 收盘后结算（不变）

### 6.5 结果输出新增内容

回测结束后，结果中新增换仓汇总数据：

- 总换仓次数
- 平均换出时浮亏百分比
- 平均 score 绝对提升量
- 完整换仓日志明细（可按日期、换出收益、score 提升等字段筛选排序）

### 6.6 参数合法性检查

以下情况应在初始化时报错并说明原因：

- `min_stop_threshold ≤ 0`
- `max_rotations_per_day < 1`
- `score_ratio_threshold < 1.0`
- `min_score_improvement < 0`
- `no_score_position_policy` 不在 skip / allow / mean 范围内
- `score_percentile_threshold` 不在 (0, 100) 范围内
- `score_min_history < 1`


## 4. 回测主引擎模块

### 4.1 功能概述

本模块是整个回测系统的主引擎，采用事件驱动 (Event-Driven) 架构，按交易日推进时间轴。其核心职责是协调数据层、策略层（买入/卖出）、资金管理层和执行层，确保各个组件在正确的时间点获取经过严格隔离的数据（防未来函数），并按 A 股有序执行"结算 → 交易 → 决策"的每日循环。

### 4.2 数据加载与预处理

#### 4.2.1 历史数据加载

- 支持读取数据目录下的 CSV 格式股票数据。
- **回溯周期**：加载数据时，必须额外加载回测开始日期之前的 lookback_days（默认 200 天）数据。确保在回测的第一天，MA60、BBI 等长期指标已有足够的历史数据进行计算。
- 在加载全量数据后，必须基于回测结束日期（End Date）进行截断。
- 在每日循环中，向策略提供数据时，必须严格执行 `df[df['date'] <= current_date]` 的操作，严禁使用 current_date 之后的数据。

#### 4.2.2 数据质量检验

- 扫描数据质量，检查在回测开始日期之前的历史数据长度，以确保常用指标可计算。如果回测开始日期当天及之前没有历史数据，输出警告；若发现大量股票（> 50%）在回测起始日的历史数据长度不足（如 <120天），输出警告日志，提示用户调整开始日期或增加回溯天数。
- 正常情况下，输出日志提示：已加载股票数量、回测开始日期、最小历史数据（开始日期之前）、最长历史数据。并输出数据长度不够计算MA60与BBI指标的股票数量。

### 4.3 策略动态加载

- **买入策略加载**：
    - 读取 `configs.json`，根据类字段动态实例化 Selector.py 中的选股器类。
    - 支持通过配置文件中的 activate 字段开启或禁用特定策略。
    - 输出日志：已加载选股器数量。
- **卖出策略加载**：
    - 根据 `sell_strategies.json` 配置字典动态实例化卖出策略（支持单一策略或组合策略，如果是组合策略应在策略名称中包含策略数量）。
    - 输出日志：已加载卖出策略的名称

### 4.4 主循环

回测引擎按交易日遍历时间轴，每日严格按照以下时序执行操作：

1. **开盘前资金结算 (T+1 Settlement)**
    
    - 调用模块，处理 T 日到账的资金（即 T-1 日卖出后冻结，T 日早晨到账的资金）。
    - 输出日志记录到账资金。
2. **执行待处理订单**
    
    - 获取 T-1 日收盘后生成的待处理订单队列。
    - 使用 T 日（当日）的市场数据（开盘价）调用执行引擎执行订单。
    - 处理成交结果（成功/失败），并输出日志记录（应包含：已执行订单明细、失败订单明细、当天执行的买入订单数量、卖出订单数量、失败订单数量、账户现金）。
3. **更新持仓状态**
    
    - 根据 T 日收盘数据，更新所有持仓的"最高价"、"持仓天数"等统计指标。
    - 如果当天没有数据，不做处理。
4. **执行卖出决策**
    
    - 遍历当前持仓，调用卖出策略。
    - 输入：截至 T 日的历史数据。
    - 输出：若触发卖出，生成 T+1 日执行的卖单（放入待执行队列）。
5. **执行买入决策**
    
    - 遍历所有激活的选股器。
    - 输入：截至 T 日的历史数据。
    - 输出：获取 T 日买入信号列表。
6. **生成买入订单**
    
    - 生成的订单标记为 T+1 日执行（放入待执行队列）。
7. **收盘后结算**
    
    - 计算并记录 T 日收盘后的账户总资产（现金 + 持仓市值），更新资产曲线。

### 4.5 信号处理与后备机制

- **处理买入信号流程**：
    1. 获取当日所有运行的选股器产生的买入信号，以 score 排序
    2. 按信号顺序依次尝试生成订单。
    3. 检查是否已满仓（`can_open_new_position`）。若已满仓，立即停止后续信号处理。
    4. 资金/持仓校验：
        - 若某信号因"资金不足"或"已持有该股"而无法生成订单，系统不应停止。
        - 系统应跳过当前失败信号，继续尝试列表中的下一个信号，直到仓位填满或信号用尽。

### 4.6 结果输出

回测结束后，需打包返回以下核心数据：

- 资产曲线：包含每日日期、总资产、现金、持仓市值的时序数据。
- 交易明细：所有已平仓交易的完整记录（开仓时间/价格、平仓时间/价格、收益率、持有天数）。
- 统计数据：最终资产、总收益率、交易总笔数、当前持仓数。

## 5. 选股器模块设计

### 5.1 功能概述

该模块定义五种内置选股器，在回测过程中负责读取股票市场的历史数据，根据选股器逻辑在股票池中进行筛选，产生买入信号并输出。支持多个选股器并行运行，系统需要汇总输出每个选股器产生的信号。

选股器在T日收盘之后运行，用于选出T+1日的买入目标。如果T日停市，则不产生任何信号。计算T日信号时，仅允许使用T日及T日之前的数据（开盘、收盘、最高价、最低价）。所有计算均使用前复权数据。

每种策略封装为独立的类，关键参数阈值设置默认值，保证可配置，参数配置统一从 `configs.json` 里调取。在回测过程中仅启用在 `configs.json` 中状态标记为已激活的选股器。

**信号评分机制**：

所有选股器在输出 BuySignal 时，必须计算并附带 score。

- **通用计算公式**：综合分 = KDJ分×40% + 量能分×30% + 动量分×20% + BBI分×10%

各指标说明:

- **KDJ分**: J值越低分数越高（0-100分区间反转）
    - J=0→100分, J=50→50分, J=100→0分
- **量能分**: 当日成交量/20日均量，比值越大分数越高
    - 量比1.0→0分, 量比3.0→100分, 线性插值
- **动量分**: 单日涨幅，最优区间为+1%到+3%
    - +2%→100分, 0%→0分, +5%→50分（避免追高）
- **BBI分**: BBI斜率，上升趋势强度
    - 斜率0.5%/天→100分, 线性缩放

### 5.2 通用指标定义

计算指标类函数应单独定义，以便在选股器类中随时调用。

**1. KDJ 计算：**

- 默认 RSV 回看周期（n）为9日，默认K值和D值平滑为3（可配置）。
- `RSV_t = (C - low_n) / (high_n - low_n) × 100`（low_n: 近n日中最低价中的最小值；high_n: 近n日中最高价中的最大值）
- K值：`K_t = (2/3) × K_t-1 + (1/3) × RSV_t`
- D值：`D_t = (2/3) × D_t-1 + (1/3) × K_t`
- J值：`J_t = 3 × K_t − 2 × D_t`
- 如果所需数据不足：
    - 如果某日之前没有数据：K、D 默认设为 50
    - 如果某日之前不足 n 日的数据：RSV按照现有数据中的最高价和最低价计算

**2. BBI 计算：**

- 标准（3、6、12、24）日均线的平均值，以收盘价计算
- 如果某日之前的数据不足24日，则该日 BBI 为空值

**3. 计算 RSV：**

- 回看周期 n 日（可配置）。
- `RSV_t = (C - low_n) / (high_n - low_n) × 100`（low_n: 近n日中最低价中的最小值；high_n: 近n日中最高价中的最大值）
- 如果某日之前数据不足 n 日：该日 RSV 为空值

**4. 计算 DIF**

- 默认快线周期为12日、慢线为26日（可配置）。
- `EMA_t = α · C_t + (1 − α) · EMA_t-1`
- 快线：`α = 2/(fast + 1)`；慢线：`α = 2/(slow + 1)`
- 如果某日之前数据不足 fast/slow 日：该日 DIF 为空值

**5. 判断 BBI 趋势好坏：**

- 从当前交易日 T 日开始往前，寻找一段 BBI 符合条件的窗口（仅需一段），约束窗口最小值与最大值，约束该窗口内允许 BBI 回撤的最大比例，窗口长度约束和回撤比例可配置。
- 设 `BBI_norm(t) = BBI(t) / BBI(T − w + 1)`，计算 `δ(t) = BBI_norm(t) − BBI_norm(t − 1)`
- 允许涨幅最低的 bbi_q_threshold %天出现 delta < 0 的情况，其他天数里 BBI 不能回撤；
- 从最长窗口`longest=min(len(bbi),max_window)`向下搜索，找到任一满足区间即通过；
- 如果 BBI 数据长度不够窗口最小值，直接返回False

**6. 寻找峰值：**

- 在最高价中寻找相对峰值

**7. 最后一次有效上穿 MA：**

- 从当前交易日往前，在最近 n 天里找最后一次有效上穿MA（前日收盘价小于均值，后日收盘价大于等于均值），返回上穿出现的位置。如果数据不足 n 天，则以该股票实际拥有的所有历史数据为准进行搜索。
- n 可配置；n 为 None 时在全历史中寻找。

**8. 知行线计算：**

- 默认知行线参数 m1=14, m2=28, m3=57, m4=114，可配置。
- 定义第一次平滑：`EMA1_t = α · Close_t + (1 − α) · EMA1_t-1`
- 定义知行短期趋势线：`zxdq_t = α · EMA1_t + (1 − α) · zxdq_t-1`
- 平滑系数 alpha：`α = 2/(10 + 1)`
- 定义各个周期的MA：`MA_i,t = (1/m_i) × Σ(Close_t-j)` for j=0 to m_i-1
- 定义知行多空线：`zxdkx_t = (MA1 + MA2 + MA3 + MA4) / 4`

**9. 判断是否通过当日过滤：**

- 涨跌幅限制、振幅限制：当日不能涨或跌超过2%，最高价与最低价差值百分比不能高于 7%。

**10. 判断是否通过知行约束：**

- 用于判断当日收盘价是否大于当日长期线所在位置、当日短期线所在位置是否大于长期线
- 标准可配置

### 5.3 内置选股策略

选股器的基本参数在定义时全部设为默认值，调用选股器时调用 `configs.json` 中的参数配置。必须满足选股器中所有筛选条件才会产生买入信号，任意一条不满足都会返回False，使该股票无法通过筛选。

- **最大回溯上限**：系统根据策略配置，确定最大所需的数据窗口长度 `L_max = max_window + Buffer (20)`。
- **滑动截取规则**：
    - 若股票历史交易天数 `N > L_max`，则仅截取最近的 `L_max` 个交易日数据参与计算（丢弃更早的历史数据）。
    - 若股票历史交易天数 `N ≤ L_max`，则全量加载所有 N 个交易日的数据参与计算。
- **后续校验**：数据加载完成后，由各个具体策略内部自行判断数据长度是否满足其最小计算要求。

#### 5.3.1 BBIKDJ Selector

**核心逻辑**：

股票整体趋势向好（BBI上升），近期收盘价波动不过大，短期趋势好于长期趋势，近期出现收盘价上穿60日均线，选取J值处于超卖区的时机入场

**基本参数（默认值）**：

- J值阈值（`j_threshold`）：-5
- J分位阈值（`j_q_threshold`): 0.1
- `bbi_min_window`: 90
- `max_window`: 90
- `price_range_pct`: 100
- `bbi_q_threshold`: 0.05

**筛选条件**：

1. 满足当日过滤
2. 收盘价波动幅度约束：近 max_window 区间中，最高收盘价与最低收盘价的差值百分比不能超过 price_range_pct；
3. BBI 持续上升：通过 BBI 趋势筛选，确保区间内只有 bbi_q_threshold 比例的天数有回撤；
4. KDJ过滤：满足当日 J 值小于 15 或当日 J 值为最近 max_window 期内最低的 j_q_threshold 范围内；如果有些日期没有有效 J 值，直接跳过；如果没有数据，返回False。
5. 上穿60日均线：当日收盘价大于 MA60；在最近 max_window 日中，存在有效上穿MA60；
6. 当日 DIF > 0
7. 满足当日知行约束 zx_condition_at_positions：收盘 > 长期，短期 > 长期

#### 5.3.2 SuperB1战法

**核心逻辑**：

在前一段周期中股票发出过B1信号，但之后一直稳定横盘，当日突然大幅下跌，并处于超卖区，近期可能出现反弹

**核心参数（默认值）**：

- `lookback_n`：60 -- 回溯窗口长度
- `close_vol_pct`：0.05 -- 收盘价区间波动率限制
- `price_drop_pct`：0.03 -- 跌幅限制
- `j_threshold`：-5
- `j_q_threshold`：0.1
- 嵌套 BBIKDJ Selector 参数配置

**参数合法性检查（报错情况）**：

1. lookback_n < 2
2. 不满足 (0 < close_vol_pct < 1)
3. 不满足 (0 < price_drop_pct < 1)
4. 不满足 (0 <= j_q_threshold <= 1)
5. B1_params 缺失

报错时应输出具体报错原因。

**筛选条件**：

1. 满足当日过滤
2. 保证数据量充足，满足有 bbi_selector.max_window + 20 + lookback_n 个交易日的历史数据。如果数据不足，返回False；
3. 搜索满足 BBIKDJ 筛选条件的 t_m 日： a. 搜索窗口为当前交易日之前的 lookback_n 日 b. 从窗口中搜索符合 BBIKDJSelector 筛选条件的日期 t_m c. 如果[t_m, T-1] 区间不足 3 天，返回False d. 要求在 [t_m, T-1] 区间中，收盘价的波动率不超过 close_vol_pct：`(Close_high - Close_low) / Close_low <= close_vol_pct`
4. 在 t_m 当日满足知行约束：收盘 > 长期，短期 > 长期
5. 当日收盘价相对前一日下跌不小于 price_drop_pct：`(Close_prev - Close_today) / Close_prev >= price_drop_pct`
6. KDJ过滤：J值小于 j_threshold 或者J值在 lookback_n 日区间内处于相对低位（属于最低的 j_q_threshold 分位内）
7. 当日知行约束仅要求满足短期 > 长期

#### 5.3.3 BBIShortLongSelector 补票战法

**核心逻辑**：

股票大趋势处于多头（BBI上升且DIF>0），长周期能量维持高位（长RSV>=threshold），近期利用短线波动带来的回撤（短RSV先处于高位，再进入超卖区，当日处在强势区）进行择时，在短线重拾动能且价格稳于长期均线之上时入场。

**核心参数**：

- `n_short`: 3 -- 短RSV计算区间
- `n_long`: 21 -- 长RSV计算区间
- `m`: 3 -- m必须大于等于2（不满足时应报错）
- `bbi_min_window`: 90
- `max_window`: 150
- `bbi_q_threshold`: 0.05
- `upper_rsv_threshold`: 75
- `lower_rsv_threshold`: 25

**筛选条件**：

1. 满足当日过滤
2. BBI上升：调用bbi_deriv_uptrend函数
3. RSV过滤 a. 最近m天，RSV_long >= upper_rsv_threshold b. 从最近m天的第一天起，存在某天满足 RSV_short >= upper，并在该日之后存在某天满足 RSV_short < lower c. 当日 RSV_short >= upper
4. MACD: DIF > 0
5. 满足知行约束：收盘>长期，短期>长期
6. 历史数据长度约束：计算 RSV 的所需长度 + BBI 最小检测窗口 + m

#### 5.3.4 PeakKDJSelector

**核心逻辑**：

股票处于上升趋势，最新峰值高于前一峰值，在价格回调至前一个有效波峰的收盘价附近、并且J值处于低位时进行买入

**核心参数**：

- `j_threshold`: -5
- `max_window`: 90 ── 回溯窗口及 J 值分位数计算周期
- `fluc_threshold`: 0.03 ── 今日收盘价与前峰收盘价的偏离度
- `j_q_threshold`: 0.10
- `gap_threshold`: 0.02 -- 峰顶和峰底的落差阈值 --> 前峰必须比中间的最低价高出一定比例

**筛选条件**：

1. 满足当日过滤
2. 找到所有峰值，基于 K 线实体顶部（`oc_max`）提取波峰，最小间距 6 天，显著性 0.5。
3. 最新波峰（Peak_t）的高度必须 > 目标前向波峰（Peak_prev）-> 高点上升
4. 如果两个波峰之间存在其他中间峰，中间峰的高度必须全部小于Peak_prev
5. 目标前向波峰与最新波峰之间，必须存在明显的低谷（前峰高度 > 区间最低收盘价 * (1 + 2%)）
6. 今日收盘价 必须落在 目标前向波峰收盘价 的 ±3% 波动范围内。
7. KDJ 过滤：当日 J 值 < 10或当日J值处于最近120天内的最低 10% 分位数）。
8. 满足知行约束：收盘 > 长期，短期 > 长期

#### 5.3.5 BigBullishVolumeSelector

**核心逻辑**：

寻找单日出现实体大阳线且伴随成交量显著放大的股票。引入知行短期趋势线进行约束，要求大涨之后的收盘价依然位于趋势线之下（或附近），以此确保入场点位于低位反弹阶段，而非高位追涨

**基本参数**：

- `up_pct_threshold`: 0.04 ── 长阳线涨幅门槛
- `upper_wick_pct_max`: 0.5 ── 上影线长度限制（上影线长度 / K线实体顶部，拒绝长上影）
- `vol_lookback_n`: 20 ── 放量比较的历史天数
- `vol_multiple`: 2.5 ── 放量倍数（当日量 > 基准均量 * 2.5）
- `min_history`: 最少历史长度（默认 = vol_lookback_n + 2)
- `require_bullish_close`: True -- 可选是否要求当日 close >= open
- `ignore_zero_volume`: True -- 计算均量时是否忽略 volume = 0
- `close_lt_zxdq_mult`: 1.0 ── 乖离率/反弹空间约束（收盘价 < zxdq * close_lt_zxdq_mult）

**参数合法性检查（报错情况）**：

1. up_pct_threshold <= 0
2. upper_wick_pct_max < 0
3. vol_lookback_n < 1
4. vol_multiple <= 0
5. close_lt_zxdq_mult <= 0

报错时应输出具体报错原因。

**筛选条件**：

1. 满足当日过滤
2. 当日阳线约束（optional）：收盘>=开盘
3. 当日涨幅 `(Close_T - Close_T-1) / Close_T-1` > up_pct_threshold
4. 上影线过滤: 上影线相对实体顶部的比例 `(High - max(open, close)) / max(open, close)` < upper_wick_pct_max
5. 成交量验证：当日成交量 必须 > vol_multiple x 过去 n 日均量。
6. 相对位置约束：当日收盘价 必须 < 知行短期趋势线 x close_lt_zxdq_mult。
7. 历史数据长度约束：max (min_history, vol_lookback_n + 2)

## 6. 卖出策略模块设计

### 6.1 功能概述

本模块定义了系统支持的所有平仓逻辑。所有策略均继承自 `SellStrategy` 基类，支持单一策略运行，也支持通过组合模式（Composite）进行"与/或"逻辑的混合调用。

### 6.2 卖出策略设计

#### 6.2.1 指标退场 (Indicator Exits)

基于技术指标形态的卖出信号，主要用于捕捉趋势反转或超买风险。

**1. KDJ 超买退场 (KDJOverboughtExit)**

- **核心逻辑**：当 J 值过高，形成超买时卖出。
- **参数配置**：
    - `j_threshold` (默认 80): J 值阈值。
    - `wait_for_turndown` (默认 False): 是否要求 当日 J < 昨日 J 才触发。
    - `use_percentile` (默认 False): 是否使用 J 值历史分位数代替绝对阈值。
    - `j_q_threshold` (默认 90): 若启用分位数，则 J 值超过历史 90% 分位时触发。
- KDJ 计算方式见（5.2 - 1）
- **触发条件**：`J > j_threshold` (或分位阈值) [如果启用 wait_for_turndown, 则还需要满足这个条件]。

**2. BBI 趋势反转 (BBIReversalExit)**

- **核心逻辑**：当 BBI 多空均线连续下跌，提示趋势由多转空时卖出。
- **参数配置**：
    - `consecutive_declines` (默认 3): BBI 连续下跌的天数。
- **规则**：
    - BBI 计算方式见（5.2 - 2）
    - BBI 数据长于 consecutive_declines
- **触发条件**：`BBI_T < BBI_T-1 < ...... < BBI_T-n`

**3. 知行线死叉 (ZXLinesCrossDownExit)**

- **核心逻辑**：当知行短期趋势线下穿多空线时卖出（与买入逻辑相反）。
- **参数配置**：无特定可调参数。
- **触发条件**：`ZXDQ_T-1 ≥ ZXDKX_T-1 and ZXDQ_T < ZXDKX_T`

**4. 均线死叉 (MADeathCrossExit)**

- **核心逻辑**：短期均线下穿长期均线。
- **参数配置**：
    - `fast_period` (默认 5): 快线周期。
    - `slow_period` (默认 20): 慢线周期。
- **触发条件**：`MA_fast,t-1 ≥ MA_slow,t-1 and MA_fast,t < MA_slow,t`。

#### 6.4.2 止盈退场 (Profit Targets)

基于盈利目标的被动止盈，锁定利润。

**5. 固定比例止盈 (FixedProfitTarget)**

- **核心逻辑**：当持仓浮盈达到预设百分比时卖出。
- **参数配置**：
    - `target_pct` (默认 0.15): 目标收益率（如 15%）。
    - `partial_exit` (默认 False): 是否分批止盈。
    - `partial_exit_pct` (默认 0.5): 若分批，卖出持仓的比例（如 50%）。
- **触发条件**：`UnrealizedPnL% ≥ target_pct`。

**6. R-Multiple 风险倍数止盈 (MultipleRExit)**

- **核心逻辑**：基于入场时的风险（R）计算止盈目标。R 通常由入场时的 ATR 决定。
- **参数配置**：
    - `r_multiple` (默认 3.0): 盈亏比目标（如赚 3 赔 1）。
    - `atr_period` (默认 14): 计算初始风险 R 时的 ATR 周期。
    - `stop_multiplier` (默认 2.0): 计算初始风险 R 的倍数（Risk=ATR×2.0）。
- **触发条件**：`CurrentProfit ≥ InitialRisk × r_multiple`。

#### 6.4.3 止损与移动止损 (Trailing Stops)

包含固定止损、移动止损及自适应止损，用于控制单笔交易风险。

**7. 百分比移动止损 (PercentageTrailingStop)**

- **核心逻辑**：价格从持仓期间最高收盘价回撤一定比例后卖出。
- **参数配置**：
    - `trailing_pct` (默认 0.08): 允许的最大回撤比例（如 8%）。
    - `activate_after_profit_pct` (默认 0.0): 仅当浮盈达到该比例后才激活止损（保护本金模式）。
- **触发条件**：`Close_t ≤ HighestClose × (1 − trailing_pct)`。

**8. ATR 移动止损 (ATRTrailingStop)**

- **核心逻辑**：利用 ATR 衡量波动率，价格跌破"最高收盘价 - N倍ATR"时卖出。
- **参数配置**：
    - `atr_period` (默认 14): ATR 周期。
    - `atr_multiplier` (默认 2.0): 安全垫倍数。
- **触发条件**：`Close_t ≤ HighestClose − (ATR × multiplier)`。

**9. 吊灯止损 (ChandelierStop)**

- **核心逻辑**：ATR 移动止损的变体，基准价格使用"持仓期间最高High价"而非收盘价，更灵敏。
- **参数配置**：
    - `lookback_period` (默认 22): 查找最高价的回溯周期。
    - `atr_period` (默认 14): ATR 周期。
    - `atr_multiplier` (默认 3.0): 倍数。
- **触发条件**：`Close_t ≤ HighestHigh − (ATR × multiplier)`。

**10. 自适应波动率止损 (AdaptiveVolatilityExit)**

- **核心逻辑**：根据当前市场波动率所处的历史分位，动态调整止损宽度。
    - 低波动期：收窄止损，保护微利。
    - 高波动期：放宽止损，防止被噪音震出。
- **参数配置**：
    - `volatility_period` (默认 20): 计算波动率的周期。
    - `lookback_period` (默认 120): 计算波动率分位的历史窗口。
    - `low/high_vol_percentile`: 判定低/高波动的分位阈值（默认 30/70）。
    - `low/normal/high_vol_stop_pct`: 不同状态下的止损比例（默认 5%/8%/12%）。
- **触发条件**：根据实时计算的 Stop% 判断 `Close_t ≤ HighestClose × (1 − DynamicStop%)`。

#### 6.4.4 放量退场 (Volume Exits)

**11. 成交量枯竭退场 (VolumeDryUpExit)**

- **核心逻辑**：当成交量连续萎缩，显著低于历史均量时卖出，视为动能衰竭或流动性风险。
- **参数配置**：
    - `volume_threshold_pct` (默认 0.5): 枯竭阈值（如低于均量的 50%）。
    - `lookback_period` (默认 20): 计算均量的基准周期。
    - `consecutive_days` (默认 3): 需连续满足的天数。
- **触发条件**：连续 3 天 `Volume_t < AvgVolume20 × 0.5`。

#### 6.4.5 持仓时间到期 (Time Based)

**12. 强制时间平仓 (TimedExit)**

- **核心逻辑**：防止资金被无效占用，到达最大持仓天数后强制平仓。
- **参数配置**：
    - `max_holding_days` (默认 60): 最大允许持仓天数。
- **触发条件**：`Position.days_held ≥ max_holding_days`。

#### 6.4.6 组合策略逻辑 (Composite Logic)

系统支持将上述单一策略组合使用，形成复杂的卖出体系。

- **配置方式**：传入策略列表及组合逻辑参数。
- **组合逻辑**：
    - **ANY (OR逻辑)**：任意一个子策略触发卖出信号，即执行卖出（如：止损 OR 止盈 OR 时间到期，满足其一即走）。
    - **ALL (AND逻辑)**：所有子策略同时触发才卖出（较少使用，通常用于极度保守的确认机制）。
